A Tale of Two Pixels

I regret I only have two Pixel 6s to trade for my sanity

Preamble

Every developer likes new shiny things, and I’m no exception – so when Google leaked announced a new phone with custom in-house silicon, an in-screen fingerprint reader, and other goodies, I looked past the hefty 6-inch screen and ordered one.

That New Phone Smell

  • first phone arrives Wed, Oct 27 (despite peoples’ fedex woes)
  • I unbox Oct 31
  • i migrate everything, set up a fingerprint…and the phone screen starts flashing on-and-off
    • after ~5 seconds, the OS crashes, restarts, lands on the lock screen, and continues
    • this isn’t a true “bootloop”, but it’s pretty damn close
  • i think to myself “huh, that’s odd”; I factory reset, setup again, and the same thing happens

Nov 1

  • contact support
  • after describing the problem, support agrees that this is a hardware issue and ships a new phone
    • narrator it wasn’t a hardware issue
  • new phone arrives Nov 4

Another Pixel 6 Enters the Arena

Nov 4

  • new phone exhibits same symptoms
  • okay so either I drew a DOA device twice, or this is a software issue.
    • if it’s a software issue, I can fix it!
  • I go into debugging mode – “what could be causing this?”
  • i do a “minimal” setup – no migration, no fingerprint and slowly start adding things back

Wisdom of the Crowds

  • in rapid succession, we we rule out:

    • a bad cable causing corrupted data during the migration (i used 2 diff cables)
    • lack of SIM card during/after setup (tried it, no difference)
    • a malformed SMS I’ve received (á la https://www.theguardian.com/technology/2015/may/27/iphone-crash-bug-text-imessage-ios)

  • Justin Talia correctly suggests to set up ADB debugging so we can tail the logs

  • watching ADB while performing normal setup operations, I see there’s a ton of errors in the logs, but this one stands out because it happens right as SystemUI died: TODO insert stacktrace here (https://app.warp.dev/block/JQ1lD9kYfgqxEXUtVojQn9)

  • GPeal comes in with the link to AOSP

  • i’m now parallelizing setups - while one phone spins, I’m poking my WiFi and Gmail credentials into the other

  • I pull a few bug reports and folks start going through them - even more errors, but no tombstone, which confirms my suspicion that this isn’t a native crash – it’s actually happening in Java code

  • (11:30pm) Kevin Barry, looking at the stacktrace, suspects Android might be expecting non-null lock screen text.

  • I’ve easily gone through 20 setups at this point, and I’m T.I.R.E.D.

  • I do a minimal setup, add lock screen text to the P6, add a fingerprint, migrate, and BOOM bootloop. So I discard this theory

  • reach out to a googler who i’m on good terms with. They aren’t on the pixel team, but I describe the issue and forward some stack traces and the bug reports

    • he agrees to send my info along

  • debugging Google’s software bugs don’t pay the bills, so I start actually working

  • in the background i’m still trying different combinations of migration parameters; none of them work

  • i accidentally discover that NOT setting the fingerprint is sort of a fix, but that’s stupid - i’d rather return the phone entirely than not have fingerprint

  • go to bed because i’m zonked

A new day, a new bootloop

  • the googler gets back to me – this is a known issue (yay) and there’s a patch coming in the next Android patch (boo!). Even better news, it’s directly caused by the lock screen text! The temporary fix is to set anything (including, but not limited to "please dear god stop crashing I just want to use my phone") as the lock screen text. I’m skeptical because I thought I tried that yesterday!

  • Giving the googler the benefit of the doubt, i try again this time, just – DING DING DING WE HAVE a winner. Fingerprint + lock screen text = Parth has a usable phone!
